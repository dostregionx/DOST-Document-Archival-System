Project Overview

The DOST Document Archival System is a full-stack web application built for the Department of Science and Technology – Misamis Oriental Provincial Office. It centralizes project management, document archiving, user administration, and geographic visualization for two primary DOST programs.

📋 SETUP Program

Small Enterprise Technology Upgrading Program. Tracks firm-level technology assistance projects from proposal through graduation.

🌿 CEST Program

Community Empowerment through Science and Technology. Manages community-level projects with multi-phase document workflows.

🗂️ Document Archival

Phase-based document upload, download, and QR tracking for physical-to-digital linking.

👥 User Administration

Role-based access control, approval workflows, and granular permission management per module.

🗺️ Geographic Mapping

Leaflet-based interactive map showing all project locations with program-based filtering.

🔔 Smart Notifications

Automated deadline alerts, approval reminders, and birthday popups with login-time toast bursts.

💬 Messaging

Internal real-time chat between staff with user mentions and read receipts.

🗄️ Multi-Database

Seamless runtime switching between CockroachDB (cloud) and PostgreSQL (local) with one-click sync.

Installation & Setup

Prerequisites

RequirementDetails

Step-by-Step Setup

  1. 1
    Clone the repository
    git clone <repository-url>
cd DOST-Document-Archival-System
  2. 2
    Install dependencies
    npm install
    The post-install script automatically downloads Windows-native LightningCSS binaries for WSL2 compatibility.
  3. 3
    Create .env file at the project root:
    DATABASE_URL="postgresql://user:pass@host:26257/defaultdb?sslmode=verify-full"
LOCAL_DATABASE_URL="postgresql://postgres:pass@localhost:5432/dost_db"
JWT_SECRET="your-long-random-secret"
EMAIL_USER="dostxmisamisoriental@gmail.com"
EMAIL_PASSWORD="your-gmail-app-password"
  4. 4
    Initialize the database
    # Cloud (CockroachDB):
npx prisma migrate deploy

# Local (PostgreSQL) — first-time:
npx prisma db push --config prisma.local.config.ts
  5. 5
    Generate Prisma client
    npx prisma generate
  6. 6
    Start the development server
    npm run dev
    Available at http://localhost:3000

Authentication & Authorization

10.1 Authentication Flow

  1. 1User submits email + password to POST /api/auth/login
  2. 2Server verifies password with bcryptjs
  3. 3On success: JWT signed with JWT_SECRET, set as HttpOnly cookie auth-token
  4. 4User object stored in localStorage (id, role, fullName, profileImageUrl)
  5. 5proxy.ts checks auth-token cookie on every request
  6. 6Unauthenticated access to protected route → redirect to /
  7. 7Authenticated access to public route → redirect to /dashboard

Database Configuration

12.1 Runtime Switching

The application supports switching databases at runtime without restarting the server. lib/prisma.ts wraps two cached Prisma clients in a JavaScript Proxy:

const prismaProxy = new Proxy({} as PrismaClient, {
  get(_target, prop) {
    const client = getClient(getActiveDb()); // reads db-config.json on every call
    const value = client[prop];
    if (typeof value === 'function') return value.bind(client);
    return value;
  },
});

12.2 db-config.json

{
  "activeDb": "local",
  "lastSync": "2026-04-26T20:00:59.143Z",
  "lastSyncDirection": "local → cloud"
}

Written by POST /api/db-config. Read on every database property access.

12.3 Data Sync

POST /api/db-sync copies all records from source to destination. It truncates destination tables and re-inserts all data. Synchronized models include: users, projects, documents, map pins, calendar events, archival records, conversations, messages, notifications, and address data.

⚠️ Sync is destructive on the destination — all existing data is replaced. Always verify source data before syncing.

File & Document Management

13.1 Document Upload Flow

  1. 1User selects file in the browser
  2. 2File read as ArrayBuffer via FileReader API
  3. 3Converted to Base64 string
  4. 4POST to /api/setup-projects/[id]/documents with { fileName, fileData, mimeType, phase, templateItemId }
  5. 5Server stores as Bytes in ProjectDocument.fileData

13.2 QR Pin Workflow

  1. 1Admin assigns a short qrPin code to a document
  2. 2QR code is generated encoding http://<LAN-IP>:3000/view-doc/[docId]
  3. 3QR code is printed and attached to the physical document
  4. 4Scanning the QR opens the public document viewer with no login required
The LAN IP is fetched from GET /api/local-ip, which uses PowerShell on WSL2 to return the true Windows network IP.

Messaging System

Data Model

User ─── ConversationParticipant ─── Conversation ─── Message

One conversation can have many participants (group or 1:1). Messages are ordered by createdAt. The lastReadAt field on ConversationParticipant tracks unread state per user.

User Mention

The UserMentionInput component detects @ in message text and queries /api/users for matching names, showing an autocomplete dropdown.

Deployment

Build for Production

npm run build   # Runs prisma generate → next build
npm start       # Starts production server

Production Checklist

  • Set all environment variables in the hosting environment
  • Run npx prisma migrate deploy against the production database
  • Ensure db-config.json contains { "activeDb": "cloud" } on the server
  • Confirm JWT_SECRET is at least 32 random characters
  • Confirm Gmail app password is set (not the account password)
  • Verify DATABASE_URL points to the production CockroachDB cluster
  • Set NODE_ENV=production
  • Test login flow end-to-end after deployment
